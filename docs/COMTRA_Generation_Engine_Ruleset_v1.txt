1. Executive Summary
This document defines the complete ruleset for Comtra’s Generation Engine — the AI-powered system that creates interface wireframes and layouts directly inside Figma, using the designer’s own design system as the single source of truth.
The Generation Engine is Comtra’s most visible differentiator. Where competitors (Figma Make, Lovable, Bolt, v0) produce output tied to default stacks (ShadCN + Tailwind) or require npm-published React libraries, Comtra generates governed interfaces from whatever exists in the designer’s Figma file: components, tokens, variables, styles, documentation, and — as of March 2026 — native Figma slots.

Core Principle: “Refine, Don’t Redraw.” Comtra never invents. It assembles. Every element in the generated output traces back to something the designer already owns — or to a curated open-source design system the designer has explicitly selected.

1.2 Cross-Function Interoperability Guarantee
FUNDAMENTAL RULE: Every element generated by the Generation Engine MUST be a standard Figma node indistinguishable from manually created design. The output MUST be fully interpretable by: (1) Figma Dev Mode for developer inspection, (2) Comtra’s Code function for production-ready code export, (3) Comtra’s Audit function for design system compliance checks, (4) Comtra’s Stats function for metrics tracking, (5) Any third-party Figma plugin or integration. The Generation Engine produces native Figma — not proprietary formats, not plugin-specific data, not locked layers.

1.3 Accessibility Disclaimer
DISCLAIMER: The Generation Engine assembles output using the designer’s own design system components and tokens. If the source design system contains accessibility issues (insufficient contrast, missing focus states, inadequate touch targets, etc.), those issues will be inherited by the generated output. It is NOT the Generation Engine’s responsibility to fix pre-existing DS accessibility problems. The Audit function is designed to detect these issues. Comtra’s recommended workflow: Audit first → fix DS issues → then Generate. When accessibility issues are detected in the source DS during generation, Comtra SHOULD display a non-blocking warning: “Your design system has [N] accessibility issues that will carry over to generated output. Run an Audit to review.”

Performance Optimization Strategies
	•	Pre-cached DS Context: On plugin launch, Comtra reads and caches the complete DS inventory. Cache refreshes on file change events, NOT on every generation request.
	•	Streaming Response: The backend SHOULD stream the JSON action plan, allowing the plugin to begin executing early actions before the full plan is received.
	•	Semantic Hashing: If the DS context hasn’t changed since last generation, the cached version is reused (same system as audit caching — cache hit = 0 extra credits).
	•	Sanitizer Protocol: Strip irrelevant Figma metadata (coordinates, selection colors, editor state) before sending to LLM. Only structurally relevant data is transmitted.
	•	Parallel Sub-tasks: For complex wireframes, Kimi’s Agent Swarm decomposes the layout into parallel sub-tasks (header, body, footer generated simultaneously).
	•	Component Reference Index: Pre-build a compact index of all available components with their variant axes, slot definitions, and accepted instances. This index is included in every generation prompt as a lookup table, not the full component tree.

3.2 Documentation as Context
This is a key differentiator. No competitor reads design documentation to inform generation. Comtra does.

	•	Component Descriptions: The .description field on components. If a Button’s description says “Use Primary for main CTAs, Secondary for supporting actions, Ghost for destructive actions in modals,” Comtra MUST respect this when choosing variants.
	•	Annotation Nodes: Text nodes prefixed “_doc/” or “_annotation/” are treated as documentation. Content is extracted and associated with the nearest parent component.
	•	Page-Level Documentation: Pages named “Documentation” or “Guidelines” are read as global generation guidance.
	•	Design System README: A frame named “README” or “DS Overview” at root level is read as the design system’s master documentation. Its text informs all generation decisions.
	•	Code Connect Metadata: If Code Connect is configured, component-to-code mappings are included as additional context to ensure generated output aligns with implementation patterns.

RULE G-DOC-1: When component documentation conflicts with generic UX patterns, the documentation ALWAYS wins. The designer’s documented intent overrides any default behavior.

4.2 DS Selection Logic
	•	Custom (Current) selected: Comtra reads the live file. All generation rules apply using the user’s own components and tokens. This is the default and recommended mode for production work.
	•	Open DS selected: Comtra loads the pre-indexed manifest for that DS. Components are instantiated from Comtra’s internal library copies. Generated output uses that DS’s tokens and component structure. The designer can then customize, adapt, or merge with their own DS.
	•	Hybrid scenario: If the user has Custom selected but their file has fewer than 10 components, Comtra MAY suggest (not force) switching to an open DS as a starting point, consistent with the Token Health Advisory System Level 2 (see Token Generation Ruleset).

IMPORTANT: When using an open DS, Comtra generates using that DS’s token values and component specs. The output will look like that design system. This is intentional — it gives designers a governed starting point they can then customize. It does NOT mix open DS components with the user’s custom components unless explicitly requested.

5. The Four Generation Modes
The Generation Engine supports four distinct modes, each triggered by the state of the designer’s context when the prompt is submitted.

5.1 Mode 1: No Selection — Create from Scratch
Trigger: No layer selected in Figma. Context Layer is empty.
Behavior: Creates entirely new wireframes on a new frame placed on the current page.

Rules
	•	G-M1-01 Frame Creation: Always create a new top-level frame with auto-layout enabled. Never scatter loose elements on the canvas.
	•	G-M1-02 Naming: The generated frame MUST have a semantic name derived from the prompt. Example: prompt “Create a desktop login page” → frame named “Login / Desktop”.
	•	G-M1-03 Placement: Place the new frame to the right of the last frame on the current page, with 100px horizontal gap. If the page is empty, place at origin (0, 0).
	•	G-M1-04 Sizing: Default to common viewport sizes: Desktop 1440×900, Tablet 768×1024, Mobile 375×812. The LLM infers the appropriate size from the prompt (e.g., “mobile login” → 375×812). If ambiguous, default to Desktop.
	•	G-M1-05 Component Inventory: Only use components that exist in the selected DS. If a prompt requests something the DS doesn’t have (e.g., “add a carousel” but no carousel component exists), Comtra MUST inform the designer: “No carousel component found in your design system. I’ll use [closest alternative] instead.”

5.2 Mode 2: Selection Active — Modify a Copy
Trigger: One or more layers are selected in Figma.
Behavior: Duplicates the selection, then modifies the copy. Originals are NEVER touched.

Rules
	•	G-M2-01 Copy First: ALWAYS clone the selected nodes before applying any modifications. The original selection MUST remain unchanged. This is a safety guarantee.
	•	G-M2-02 Copy Naming: Append “ — Generated” to the duplicated frame’s name. Example: “Login / Desktop” → “Login / Desktop — Generated”.
	•	G-M2-03 Copy Placement: Place the copy directly to the right of the original, with 100px horizontal gap.
	•	G-M2-04 Preserve Structure: Maintain the original’s auto-layout structure, layer hierarchy, and component bindings. Only modify what the prompt explicitly requests.
	•	G-M2-05 Scope Awareness: If the selection is a single component instance, modifications are scoped to that instance. If the selection is a frame containing multiple elements, the entire composition is duplicated and modified.

5.3 Mode 3: Upload Screenshot — Pixel-to-DS Conversion
Trigger: Designer uploads an image via the Upload Image area.
Behavior: AI analyzes the screenshot and reconstructs the interface using the selected design system’s components and tokens.

Rules
	•	G-M3-01 Visual Analysis: Kimi K2.5’s multimodal vision analyzes the screenshot to identify: layout structure (grid, sections, hierarchy), UI elements (buttons, inputs, cards, navigation), approximate spacing and proportions, color relationships, typography scale.
	•	G-M3-02 DS Mapping: Every identified element MUST be mapped to the closest available component in the selected DS. Kimi does NOT generate custom shapes or raw rectangles to “mimic” the screenshot — it uses real components.
	•	G-M3-03 Fidelity Spectrum: The output is a governed interpretation, not a pixel-perfect copy. Structure and hierarchy are preserved; visual treatment comes from the selected DS’s tokens.
	•	G-M3-04 Unmapped Elements: If the screenshot contains elements with no DS equivalent, Comtra lists them in the response: “The following elements from your screenshot had no match in your DS: [carousel, rating stars, custom illustration]. These were approximated with [alternatives used].”
	•	G-M3-05 Multi-Platform: Screenshots from any platform (web, iOS, Android, desktop apps) are accepted. The output is always built with the selected DS, regardless of the screenshot’s source platform.

5.4 Mode 4: Paste Link — Figma Reference
Trigger: Designer pastes a Figma wireframe link in the prompt.
Behavior: Comtra fetches the linked frame’s structure (via Figma API) and uses it as a reference for generation.

Rules
	•	G-M4-01 Link Resolution: The pasted link is resolved to a specific Figma node via the REST API. Comtra reads the node’s structure, components, and layout.
	•	G-M4-02 Reference, Not Copy: The linked frame is a reference for layout and structure. Comtra rebuilds it using the CURRENT file’s DS, not the source file’s components.
	•	G-M4-03 Cross-File Support: Links can point to frames in other Figma files, enabling designers to reference team libraries, inspiration files, or competitor analysis.
	•	G-M4-04 Permission Check: Comtra MUST verify the user has at least view access to the linked file. If access is denied, display: “Can’t access the linked file. Make sure you have view permissions.”
	•	G-M4-05 Prompt Augmentation: The designer can combine a link with natural language: “Recreate this [link] but make it mobile-first with a bottom navigation.” The link provides structure; the prompt provides intent.

6. Governance Rules
These rules are the core of what makes Comtra’s generation different from every competitor. They ensure that every generated output is structurally sound, DS-compliant, and production-ready.

6.1 Component Governance
	•	G-GOV-01 No Invented Components: The Generation Engine MUST NEVER create raw shapes, rectangles, or custom elements to represent UI components. Every visible element in the output MUST be either: (a) an instance of a component from the selected DS, or (b) a native Figma element justified by layout needs (auto-layout frame, divider line).
	•	G-GOV-02 Correct Variant Selection: When a component has variants, the engine MUST select the contextually appropriate variant. A disabled button in a form with validation errors. A filled input in an active form state. The selection MUST be informed by: (1) component documentation, (2) prompt context, (3) UX best practices.
	•	G-GOV-03 Slot Population: When a component has slots (Figma native slots, as of March 2026), the Generation Engine MUST populate them with appropriate content. Empty slots are acceptable only when the component’s documentation explicitly defines an “empty” state.
	•	G-GOV-04 Instance Integrity: All generated component instances MUST remain connected to their main component. NEVER detach instances. If customization is needed that would require detaching, the engine MUST use component properties (overrides, slot content, variant switching) instead.
	•	G-GOV-05 Component Documentation Compliance: If a component’s description includes usage guidelines, the engine MUST follow them. Example: if a Modal component’s description says “Always include a close button and at least one action,” the generated modal MUST have both.

6.2 Token Governance
	•	G-GOV-06 No Hardcoded Values: Every color, spacing value, corner radius, and font size in the output MUST reference a design token (Figma variable). Hex colors, pixel values, and raw font sizes are STRICTLY PROHIBITED.
	•	G-GOV-07 Semantic Token Priority: When both primitive and semantic tokens exist, ALWAYS use the semantic token. Example: use “color/surface/primary” not “color/blue/500”. This ensures theme compatibility (light/dark mode switching).
	•	G-GOV-08 Mode Awareness: If the DS has multiple variable modes (light/dark, brand A/brand B), the generated output MUST work correctly across all modes. This means only using tokens that are defined in all modes.
	•	G-GOV-09 Spacing System Compliance: All spacing values MUST come from the DS’s spacing scale. If the DS uses an 8px grid (8, 16, 24, 32...), the engine MUST NOT insert 10px or 14px gaps. Auto-layout gap values are set via spacing tokens.

6.3 Layout Governance
	•	G-GOV-10 Auto-Layout Mandatory: Every generated frame MUST use Figma auto-layout. Absolute positioning is PROHIBITED except for overlay elements (modals, tooltips, toasts) which require it by nature.
	•	G-GOV-11 Responsive Intent: Generated layouts MUST express responsive intent through auto-layout properties: fill/hug, min/max width, wrap behavior. The output should adapt gracefully when the designer resizes the frame.
	•	G-GOV-12 Hierarchy Depth: Maximum nesting depth is 6 levels. Deeper nesting indicates structural problems. If the layout requires >6 levels, the engine MUST decompose into sub-components or sections.
	•	G-GOV-13 Semantic Layer Naming: Every layer MUST have a semantic name. NEVER use Figma defaults (“Frame 43”, “Rectangle 12”). Naming follows the DS’s convention if documented, or defaults to: Section/Function (e.g., “Header/Navigation”, “Form/Email Input”, “Actions/Submit Button”).
	•	G-GOV-14 Slots for Composition: When building container components (cards, modals, sections), the engine SHOULD leverage native Figma slots to create flexible, composable structures. This ensures the generated output matches production code patterns where slots/children/render props are used.

7. Variability Engine
A critical requirement: the Generation Engine MUST NOT produce the same output repeatedly. Every generation must feel fresh, intentional, and contextually unique — even for similar prompts.

ANTI-PATTERN: The engine generates the same login page layout every time someone asks for a login page. Identical element order, identical spacing, identical component choices. This is unacceptable.

7.1 Variability Strategies
	•	G-VAR-01 Layout Variation: For the same prompt type, the engine MUST rotate between at least 3 distinct layout approaches. Example for “login page”: (A) centered card with illustration left, (B) full-bleed split screen, (C) minimal centered form with background pattern. The choice is seeded by a randomized variability parameter.
	•	G-VAR-02 Component Variation: When multiple DS components serve the same function (e.g., TextInput and SearchInput both accept text), the engine SHOULD vary which it uses based on context appropriateness and previous generation history.
	•	G-VAR-03 Content Variation: Placeholder text MUST NOT be generic lorem ipsum. The engine generates contextually appropriate placeholder content. A login page gets “Email address” and “Password”, not “Label” and “Placeholder”. An e-commerce page gets realistic product names and prices.
	•	G-VAR-04 Spacing Variation: Within the DS’s spacing scale, the engine varies density. Some generations are airy (using larger spacing tokens), others are compact. This is influenced by the prompt (“clean and spacious” vs “data-dense dashboard”).
	•	G-VAR-05 Composition Variation: The engine varies how it composes sections: vertical stack vs. grid layout, single column vs. multi-column, fixed header vs. sticky sidebar. This prevents structural monotony.

7.2 Variability Seed System
Each generation request includes a variability_seed (random integer 0–9999) that influences layout, spacing, and composition decisions. The seed is included in the LLM prompt to ensure deterministic-but-varied output. Designers CAN request a specific seed to reproduce a generation (“generate again with the same layout”), or explicitly request a new variation (“try a different approach”).

7.3 Smart Defaults with Documentation Override
The engine maintains a library of UX pattern defaults (common layouts for login, dashboard, settings, onboarding, etc.) that serve as starting points. These defaults are ALWAYS overridden by: (1) the designer’s prompt, (2) DS documentation, (3) component-level usage guidelines. The defaults exist only to provide intelligent structure when the prompt is vague.

8. Output Standards
This section defines what “pulito” (clean) output means in concrete, measurable terms.

8.1 Figma-Native Output Requirements
	•	G-OUT-01 Auto-Layout Throughout: Every frame, section, and content group uses auto-layout with correct direction, gap (from spacing tokens), and padding (from spacing tokens).
	•	G-OUT-02 Variables Bound: Every fill, stroke, effect, spacing value, corner radius, and font size is bound to a Figma variable. The Dev Mode inspector MUST show variable names, not raw values.
	•	G-OUT-03 Component Instances: All UI elements are proper component instances, fully connected to main components. No detached instances. No raw shapes mimicking components.
	•	G-OUT-04 Slots Populated: Where components have native Figma slots, they are populated with appropriate content instances. Slot constraints (accepted instance types) are respected.
	•	G-OUT-05 Layer Names: Every layer has a semantic name following the DS’s naming convention (or Comtra’s default convention if none is documented).
	•	G-OUT-06 Layer Order: Layers are ordered logically in the Layers panel: top → bottom matches visual top → bottom (consistent with auto-layout direction).
	•	G-OUT-07 No Orphan Nodes: No hidden layers, no zero-opacity elements, no off-canvas nodes. Every element in the generated output is visible and intentional.
	•	G-OUT-08 Text Content: All text nodes contain contextually appropriate content (never “Label”, “Text”, or “Lorem ipsum”). Text styles are applied via text style variables or typography tokens.

8.2 Dev Mode Readiness
Generated output MUST be fully inspectable in Figma’s Dev Mode:
	•	Variables visible: Developers see token names (e.g., “spacing/md”) not raw values (e.g., “16px”).
	•	Component names visible: Developers see component paths (e.g., “Button / Primary / Large”).
	•	Auto-layout properties visible: Developers see layout direction, gap, padding, alignment.
	•	Slot structure visible: Developers see slot boundaries and accepted types, directly mapping to code slot/children patterns.
	•	Code Connect compatible: If Code Connect is configured, generated component instances MUST match their connected code components.

8.3 Code Export Readiness
Generated output MUST be exportable via Comtra’s Code function (and the Token Generation Ruleset) without any manual cleanup:
	•	Clean CSS/JSON token references: Every variable binding exports as a proper custom property reference or JSON token path.
	•	Semantic HTML inference: The layer hierarchy and naming enable correct semantic HTML generation (nav, main, section, article, form, button).
	•	Accessible markup: Component instances with documented ARIA roles export with correct role attributes.

10.3 Validation Rules
The backend MUST validate the action plan before sending it to the plugin:
	•	V-GEN-01: Every componentKey in INSTANCE_COMPONENT actions MUST exist in the selected DS’s component index. Hallucinated component names are rejected.
	•	V-GEN-02: Every variableId in SET_VARIABLE actions MUST exist in the DS’s variable index. Invented tokens are rejected.
	•	V-GEN-03: Every variantProperties combination in INSTANCE_COMPONENT MUST be a valid combination for that component. Invalid variant combos are rejected.
	•	V-GEN-04: POPULATE_SLOT actions MUST reference slot names that exist on the target component, and the content MUST be an accepted instance type for that slot.
	•	V-GEN-05: The total action count MUST be proportional to the complexity tier. A “micro” generation with 50 actions indicates the LLM overproduced and SHOULD be re-requested with tighter constraints.
	•	V-GEN-06: No action may set a raw color value (hex), raw spacing value (px), or raw font size. All visual properties MUST reference variables.

12. Implementation Notes for Antigravity
12.1 Open DS Library Architecture
Each open design system (Material Design 3, iOS HIG, Ant Design, Carbon, Bootstrap 5, Salesforce Lightning, Uber Base Web) is stored as a pre-indexed JSON manifest containing: component inventory with variant axes and slot definitions, token map with all variables and their values across modes, documentation extracts (component descriptions, usage guidelines), and a compact reference index optimized for LLM context (under 4K tokens per DS).
These manifests are stored server-side and loaded on DS selection change. They are NOT bundled with the plugin. The plugin downloads the relevant manifest on first use and caches it locally for the duration of the session.

12.2 Kimi K2.5 Prompt Engineering
The generation prompt sent to Kimi MUST include: (1) the system prompt with generation rules (extracted from this ruleset), (2) the DS component reference index, (3) the DS documentation context, (4) the variability seed, (5) the designer’s prompt, (6) mode-specific context (selection data, screenshot base64, or link-resolved structure). The system prompt MUST enforce JSON-only output with the Action Plan Schema defined in Section 10.

12.3 Figma Plugin API Execution
The plugin executes the action plan using standard Figma Plugin API calls. Key methods: figma.createFrame() for frame creation, figma.getNodeById() for component lookup, component.createInstance() for instancing, instance.setProperties() for variant selection, node.setBoundVariable() for token binding, and the new slot APIs for native slot population. All operations are wrapped in a single figma.commitUndo() block so the entire generation can be undone with Cmd+Z.

12.4 Error Handling
	•	LLM Timeout: If Kimi doesn’t respond within 8 seconds, retry once. If retry fails, fall back to Claude Sonnet 4.5.
	•	Validation Failure: If the action plan fails validation (hallucinated components, invalid tokens), re-request with a tighter prompt that explicitly lists available components.
	•	Partial Execution: If the plugin encounters an error mid-execution (e.g., a component instance fails), it MUST roll back the entire generation (undo all created nodes) and display the error. No partial output is shown.
	•	Credit Protection: Credits are deducted ONLY after successful canvas render. Failed generations cost the user nothing.
